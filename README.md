# badge-maker-rs

`badge-maker-rs` is an experimental Rust project that aims to match the final
rendered output of [Shields.io `badge-maker`](https://github.com/badges/shields/tree/master/badge-maker)
as closely as possible, and eventually mature into a standalone Rust crate.

The repository already contains a working Rust renderer covering all five
official `badge-maker` styles, with upstream-backed regression tests guarding
both rendering behavior and final raster output.

The immediate motivation for this work is [deps.rs issue #245](https://github.com/deps-rs/deps.rs/issues/245).
The goal is for badges generated by deps.rs to look and behave like Shields.io
badges, rather than continuing to drift through a separately maintained custom
implementation.

## Project Status

The project is still experimental, but the core rendering pipeline is already in
place:

- Final raster output has higher priority than byte-for-byte SVG parity.
- The first `anafanafo` compatibility pass is in place, using upstream width
  tables compiled into Rust static data at build time instead of runtime JSON
  parsing.
- Bun-based upstream reference runners are checked into the repository so visual
  compatibility can be validated without keeping a persistent `node_modules/`
  directory in the worktree.
- All five official `badge-maker` styles are covered by the current renderer and
  protected by upstream comparison tests.
- The public API has been deliberately narrowed to a Rust-first
  `BadgeOptions::new(message)` + `make_badge` surface instead of mirroring the
  JavaScript package shape.
- Rustdoc style preview SVGs are tracked in the repository and verified against
  the current renderer output.
- `cargo doc --no-deps` is enough to generate local documentation previews with
  working images: `build.rs` inlines the tracked SVG previews as `data:` URLs,
  which also keeps the output compatible with docs.rs.

## Testing

Run the full suite with `just test`.

The test strategy is intentionally integration-heavy and upstream-driven rather
than centered on fine-grained unit tests.

The current checks are:

- Reference execution through the upstream `badge-maker` npm package via Bun.
- Pixel-by-pixel raster comparison between upstream SVG output and Rust SVG
  output.
- Regression tests for public Rust API semantics.
- Verification that tracked rustdoc style preview SVGs still match the current
  renderer output.

This combination is meant to keep the main truth source aligned with the
project's goal: final rendered image compatibility. Pixel comparison is the
primary gate; SVG source inspection is used only where it is still useful for
diagnostics or public API coverage.

For stable data-driven modules such as `anafanafo`, the repository also avoids
leaving unnecessary cost in the runtime path:

- Upstream width-table JSON files remain checked in for provenance and review.
- `build.rs` converts those JSON files into Rust static tables during the build.
- Runtime measurement consumes the generated static tables directly.

## Public API

The crate root currently exposes only the types and functions directly involved
in rendering:

- `BadgeOptions`
- `Color`
- `NamedColor`
- `Style`
- `Error`
- `make_badge`

Minimal usage looks like this:

```rust
use badge_maker_rs::{BadgeOptions, Color, Style, make_badge};

let svg = make_badge(
    &BadgeOptions::new("passing")
        .label("build")
        .color("brightgreen".parse()?)
        .style(Style::Flat)
        .build(),
)?;
assert!(svg.starts_with("<svg "));
# Ok::<(), Box<dyn std::error::Error>>(())
```

The compatibility boundary is intentionally explicit:

- The target is rendered output compatibility, not full Node.js package API
  parity.
- Upstream `badge-maker` remains the reference implementation for rendering
  behavior.
- The crate does not expose `ValidationError`, object-shape validation wrappers,
  JSON output, or other Node-specific entry points.
- Fields such as `logo_width`, when present, are treated as Rust-side
  convenience inputs rather than compatibility promises for the JavaScript API.

Current input semantics:

- `label` and `message` are trimmed before layout.
- `message` is the only required input and is provided through
  `BadgeOptions::new(message)`.
- Text and attribute content are XML-escaped before SVG emission.
- The recommended typed path for everyday use is
  `"brightgreen".parse::<Color>()`.
- Omitted `color` / `label_color` follow Shields defaults.
- Invalid `Color::literal(...)` input does not error; it falls back to style
  defaults.
- `left_link` / `right_link` encode the link structure directly:
  `left_link` alone wraps the full badge body, `right_link` alone links only the
  right half, and both together create independent left/right links.
- `logo_data_url` corresponds to `badge-maker`'s `logoBase64`.
- `logo_width` corresponds to `badge-maker`'s `logoWidth`.
- `id_suffix` corresponds to `badge-maker`'s `idSuffix`.
- `id_suffix` is currently the only public input that returns a dedicated error.

## Breaking Changes

- `BadgeOptions.links` was removed in favor of `left_link` / `right_link`.
- `BadgeOptions::set_links(...)` was removed in favor of setting
  `left_link` / `right_link` directly.
- `BadgeOptions.logo_base64` was renamed to `logo_data_url`.
